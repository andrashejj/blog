I gave red teaming one paragraph in the [remedies post](/blog/remedies-for-the-slop) and moved on. It deserved more, because once agents write a real share of the code it's one of the few practices that still keeps a human in the part of the loop where judgment lives: one named person who assumes the system is already broken and goes looking for the break.

A reviewer asks "is this correct?" and usually answers yes, because agent-written code is very good at looking correct. A red teamer asks "how does this hurt us?" That question finds different things, and the rest of this post is about the things it finds.

## Why the agent makes this harder

The old failure mode was a tired human making a mistake, and another human reading carefully might catch the off-by-one or the missing null check. Review worked, some of the time, because both sides were fallible in the same human ways.

The new failure mode doesn't look like a mistake. The agent produces confident code at volume, wrapped in tests that pass and a commit message explaining why the approach is sound. The fluency is the whole problem. When the code reads right and ships with its own justification attached, the natural response is to believe it: you skim, it's plausible, you approve. The agent has written both the answer and the argument for why you can stop checking.

And here's the part that catches people. The tests pass *because* they avoid the dangerous case, not despite it. An agent writes the test it can imagine passing, which is the same shape as the code it just wrote. The test and the bug agree with each other. Green doesn't mean safe; it means the code is consistent with a test that shares its blind spot.

## A worked example

Abstract advice about "thinking adversarially" evaporates the first sprint someone's behind, so let me make it concrete. Say the agent adds a Stripe webhook handler. The PR is eighty lines, the tests are green, the commit message says "Handle payment confirmation webhook." Here's roughly what it wrote:

<Quote>
POST /webhooks/stripe → parse the JSON body, read event.type, and on `payment_intent.created` look up the order by metadata.order_id and set status = paid. Returns 200.
</Quote>

A reviewer reads that and sees a clean handler with a passing test. A red teamer reads it for twenty minutes and finds three separate ways it hurts you.

First, it never verifies the signature. Stripe signs every webhook with a secret and expects you to check the `Stripe-Signature` header before trusting a byte of the body. This handler doesn't, which means anyone who learns the URL can POST a fake `order_id` and mark any order paid. The reason the agent skipped it is worth sitting with: the test constructs its own fake event object and calls the handler directly, so the test never needed a valid signature, so nothing in the loop ever demanded one. The missing check and the passing test are the same fact.

Second, it fires on the wrong event. `payment_intent.created` means Stripe *started* a payment, not that money moved. The event you want is `payment_intent.succeeded`. The handler is internally consistent and does exactly what the code says; it just confirms payments that haven't happened. No automated check catches this, because "wrong event name, right structure" is invisible to a type system and to a test written by the same author who picked the wrong event.

Third, it isn't idempotent. Stripe retries webhooks, sometimes for days, until it gets a 200. Nothing here records that an event was already processed, so a retried `succeeded` re-runs the fulfilment: a second shipment, a second license, a second email. The happy path the agent built runs fine exactly once, which is exactly how often the test runs it.

None of these are exotic. They're the inputs the agent didn't imagine, the checks it didn't add, and the gap between what you said and what you meant. That's the whole job, and the example maps onto the three fronts you attack on every change.

## The three fronts

Inputs it didn't imagine. The agent tested the cases it thought of, so your time goes to the ones it didn't: the empty list, the Unicode in the name field, the timezone at the date boundary, the request that arrives twice because the client retried. Agents are strong in the middle of the input space and careless at the edges. Live at the edges.

The thing it didn't build. Read the change for what's absent, not just what's present. The rate limit nobody put on the new endpoint. The signature check missing from the webhook. The timeout absent from the outbound call that will one day hang and take the connection pool with it. Absences don't appear in a diff; you can't review a line that isn't there, so someone has to hold the shape of the whole system in their head and notice the missing piece. This is the front tools are worst at and the one that takes systems down.

Intent against behavior. Code that's consistent and does the wrong thing passes everything. The agent optimized for what you typed; the red teamer checks against what you meant, which usually lives in a human's head and nowhere in the repository. The `created`-versus-`succeeded` bug is this front in one line.

## Running it so it sticks

Rotate the role. One person per risky change, named, whose only job for that session is to break it. Never the author, because adversarial review of your own work is theatre: you'll defend the choices you already made. Rotating it also spreads the skill, and thinking like an attacker is learnable by people who've never once been asked to.

Time-box it and write it down. Twenty focused minutes on "how does this fail?" beats an hour of polite line-by-line reading. The output is a short note on the PR, and it's worth keeping in a fixed shape: what I attacked, what I found, what we're choosing to accept. For the webhook that note reads: *attacked signature, event type, retries. Found no signature verification (blocking), wrong event, no idempotency. Accepting nothing, all three get fixed before merge.* When a note instead says "we aren't covered here and we've decided that's fine for now," that's a real decision a named person made on purpose, and you can find it again when it bites.

Keep the surface small enough to attack at all. You cannot red team a two-thousand-line diff. Nobody can hold it in their head, so the adversarial reading degrades into skimming, which is the exact failure you were trying to prevent. Small PRs earn their keep here more than anywhere else: they're the thing that lets a human stay in the loop.

## It isn't a security review

Half the webhook findings were security bugs and one wasn't, and that's the point. Red teaming isn't a pass with a scanner or a checklist of the OWASP top ten, though it'll surface plenty of those. It's broader and it's cheaper: one person asking how a specific change fails under an adversary, where the adversary might be a real attacker, a retrying client, a clock, or just reality declining to match the happy path. The wrong-event bug had no attacker and would still have refunded money to people who never paid. A scanner finds none of that. A person holding the goal in their head finds all of it.

## The part that's yours

The machine produces the work, the tests, and the confidence. It can't be the one looking for the lie, because it doesn't know it's lying and has no stake in being wrong. Red teaming is where a human decides that "it looks fine" and "it is fine" are different claims and goes to find out which is true.

That used to happen a little by accident, in the friction of writing code by hand and noticing things along the way. The friction is gone, so you arrange the noticing on purpose: give it to a named person, time-box it, write down what they found, and make it part of how risky changes ship. Teams that do this still move fast. They've just stopped confusing speed with safety.
