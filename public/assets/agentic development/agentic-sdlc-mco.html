§1Why this proposal

Five observations about engineering work after agents.

  1. Agent leverage is asymmetric. Senior engineers become 10x more productive with agent orchestration; the best approach 100x. Less-skilled agent use produces volume but not results, and adds review work for the best engineers. The operating model has to concentrate leverage on senior engineers (Architects, FDEs), not distribute it evenly.
  2. PR review doesn't scale with the volume of agent-produced code. The bottleneck moves from authoring to verifying.
  3. PR-by-PR review catches local issues, not system-level ones. Cross-team patterns stay invisible at PR-grain: duplicated abstractions, drift from architecture intent, inconsistent error handling.
  4. The apprenticeship loop is broken. Junior learning relied on writing production code; agents do that now. The loop needs redesign around review, planning, and architecture.
  5. The pre-agent team model doesn't map to the new work. PO + QA + EM + engineers assumes humans write every line. Work has shifted; the role structure has to follow.

§2Teams and roles

Engineering organises around scrum modules: groups of 3–5 scrum teams that together cover a product area or domain. No generic teams. Each team in a module has a specialty. Alongside the modules sits the platform team: Engineering Catalysts, built from the existing DevOps team. AI Core remains separate, with strong ties to Engineering Catalysts.

Scrum modules · 3–5 specialized teams each

Across the org, ~20 scrum teams organise into 5–7 modules. Inside each module, at least one team of each type below.

Customer-embedded

Forward Deployed scrum team

Led by a Forward Deployed Engineer (FDE): a senior engineer merged with a product owner. Builds customer-specific configurations and integrations around the monolith (BEKB and similar). Routes shared-code changes through the other teams in the module.

1 per module
Greenfield · novel work

Spike team

Heavy lifting on new things. Difficult algorithms, novel architecture, exploratory work. Ships the first working version; risk-tolerant by design. Hands stabilised code to the Optimization team.

1 per module
Hardening · refactor

Optimization team

Algorithm optimization, refactoring, performance, hardening. Takes ownership of Spike team output once it stabilises. The "make it actually good" team.

1 per module

Each scrum team, whatever its specialty, is 3–4 people: Architect Engineer, Integration Engineer(s), SME. Significantly leaner than today's ~7-person scrum teams, where QA, PO, and EM roles are absorbed or removed. Modules with 4 or 5 teams add a second Spike or Optimization team depending on where the workload sits.

§3What changes

L0/L1/L2 planning and agents writing code from L2 are already in place. The proposal adds review discipline, branch discipline, quality structure, and lifecycle ownership around them. The principle: the goal is work that's verified, shipped, and owned, not raw output volume.

Planning review and challenge

Each level of the existing hierarchy becomes a real gate:

  • L0 (SME-authored): reviewed within the team
  • L1 (Architect-led): challenged by red team if blast radius warrants
  • L2: reviewed against L1 before agents execute
  • PR: reviewed against L1 intent, not just for code correctness

Designing and challenging L2 well is the new craft.

Branch and PR discipline

  • Story branches → feature branches → DEV
  • PRs under 100 LOC, reviewed in-team
  • ≤5 feature-sized PRs per team per 2-week sprint into DEV
  • DEV stays production-ready; broken DEV stops the line

Quality · defense in depth

Four layers, human to mechanical:

  1. Red teaming: monthly half-day (~4h), run by the Engineering Director with 5–6 senior panelists (Architects, FDEs, Engineering Catalysts lead). Pre-read is L0 and L1 only; the point is system-level review, not code review. Default opening: "If this system fails in production six months from now, what is the most likely cause, and would we be able to detect it?" Findings tracked; closure target 30 days. Any PR can be tagged for red team review on demand, or via random cross-team sampling each sprint.
  2. Continuous evaluation: system and agent evals block merge on regression
  3. Observability by default: required in every L1; no observability plan, no L1 approval
  4. Progressive delivery: shadow → canary → full for shared components

Lifecycle ownership

  • Scrum modules: design → ship + feature behavior in production
  • Forward Deployed scrum teams: customer-specific deployments + incidents
  • Engineering Catalysts: platform up to DEV + 3rd-level support
  • AI Core: agentic tooling layer (Agentic Planning Solution, Implementation Agent, evals)
  • Operations: PROD
  • Engineering Director + Architects: program governance (red team, certification, methods, ADLC standards)

Operating rhythm

The SDLC runs on a 2-week sprint cadence across all scrum modules. Daily standups are extended to include agent status. Planning, demos, and retros follow standard scrum timing. Red team runs monthly half-day.

Full operational detail (formats, checklists, rationale) in the companion THINK Rituals + Guides doc (think-program-rituals.html).
