§1Daily Thinkup · the extended standup

Purpose

Quick sync, agent status check, and pattern spotting across the team. The "extended" part: each engineer's update includes their agent status (what's running, what's blocked, what needs review). The "think" part: the Architect spots cross-PR or cross-engineer patterns, not just collects status.

Format

  • Duration: 15–20 minutes, hard cap
  • Mode: async-first (written posts in a dedicated channel) by default; sync only when needed
  • Cadence: every working day
  • Participants: full scrum team (3–4 people including Architect, Integration Engineer(s), SME)

Per-engineer update format

  • Shipped yesterday: PR links, brief outcome
  • Doing today: 1–2 lines
  • Blockers / needs review: explicit asks, with links
  • Agent status: what's running, what's blocked, what needs human review

Architect's job in standup

Watch for patterns: two engineers solving the same problem in parallel, drift from L1 intent, duplicated agent prompts that suggest a shared utility. Note them and route them to dedicated review sessions, don't solve them in standup.

Before standup · checklist

  • ☐ Posted your async update (if async)
  • ☐ Linked any PRs awaiting review
  • ☐ Listed any agents currently running with a one-line description
  • ☐ Flagged blockers explicitly

During standup · checklist

  • ☐ Each engineer gives their update (≤2 min each)
  • ☐ Architect captures patterns silently or in a side note
  • ☐ Blockers acknowledged and routed (not solved here)
  • ☐ End on time

What NOT to do

  • Don't problem-solve. Pattern: someone says they're blocked. Response: "Let's discuss after standup." Not: "Have you tried X?"
  • Don't skip agent status. If an agent is running unattended, the team needs to know.
  • Don't run over 20 min. If it's running long, the format is broken. Fix it in retro.
  • Don't use standup for design discussions or red team prep. Those have their own sessions.

Why

The standup is the cheapest place to spot drift. When agents write code in parallel across the team, you need a daily sync point that catches divergence before it lands in DEV. The agent-status section is what makes standup work for the agentic SDLC.

§2Red Team Session

Purpose

System-level review. PR-by-PR review catches local issues; red team catches what they miss: architecture drift, cross-system risks, regulator-visible issues, blast-radius assumptions that look fine in isolation but compound at scale.

Cadence

Monthly half-day session (~4 hours). On-demand additional sessions for high-risk work (regulator filings, customer-facing launches, multi-team migrations).

Panel composition

  • Chair: Engineering Director
  • Panelists (5–6 total): 2–3 Architect Engineers from different modules; 1 Forward Deployed Engineer; Engineering Catalysts lead (Founding Engineer); optionally the SME on the work being reviewed
  • Note-taker: assigned per session, rotates

Pre-read

  • L0 (intent) document
  • L1 (system design) document
  • NOT code. The session is about intent and design. Code review happens in PRs.

Default opening question

"If this system fails in production six months from now, what is the most likely cause, and would we be able to detect it?"

Sample follow-up questions

  • What's the worst path through this system?
  • What assumptions are baked in that we don't notice?
  • Where would a malicious user push hardest?
  • Where would a regulator's auditor look first?
  • What happens at 10x current load? At 100x?
  • What's the rollback story?

Before session · checklist

  • ☐ L0 + L1 docs circulated at least 48h ahead
  • ☐ Panel invited and confirmed
  • ☐ Note-taker assigned
  • ☐ Session goal stated in the invite
  • ☐ Owning Architect prepared to walk through L0 + L1

During session · checklist

  • ☐ Owning Architect walks the panel through L0 + L1 (15 min)
  • ☐ Chair opens with the default question
  • ☐ Panel surfaces concerns; chair ensures every panelist contributes
  • ☐ Concerns ranked: high / medium / low
  • ☐ Each concern gets an owner and closure date (target 30 days)

After session · checklist

  • ☐ Findings document published within 48h
  • ☐ Each owner has accepted
  • ☐ Tracking system updated
  • ☐ Follow-up scheduled at 30 days to verify closure

When to tag a PR for ad-hoc red team

  • Touches shared components (high blast radius)
  • Crosses module boundaries
  • Touches regulated data
  • Architect requests it
  • Random sampling each sprint (1–2 PRs per module)

Why

System-level failures kill banks, and PR review at 100 LOC granularity won't catch them. The monthly half-day is cheap insurance against the issues that show up in regulator audits and post-mortems.

§3Branching Strategy · when, what, why

The flow

    SB1["Story branch"] -->|"PR ≤100 LOC
team review <4h"| FB["Feature / epic branch"]
    SB2["Story branch"] --> FB
    SB3["Story branch"] --> FB
    FB -->|"≤5 features/sprint/team"| DEV["DEV
always production-ready"]
    DEV -->|"shadow → canary → full"| PROD["Production"]

The rules

  1. DEV is always production-ready. Broken DEV stops the line.
  2. PRs into feature branches are ≤100 LOC and reviewed in-team within 4 working hours.
  3. Max 5 feature-sized PRs into DEV per team per 2-week sprint.
  4. Story branches hold single concerns. One story, one branch, one PR.
  5. Architect approval needed for exceptions. No silent rule-bending.
  6. Any PR can be tagged for red team review on demand or via random sampling each sprint.

When to use what · decision tree

  • Single small change (≤100 LOC, single concern): story branch → feature branch (or directly to DEV if trivial and pre-approved)
  • Multi-step feature: many story branches → feature branch → DEV (when feature complete)
  • Hotfix: story branch → feature branch with expedited review path
  • Refactor that touches shared components: opt-in to red team review first, then standard branching
  • Spike / exploratory: story branch on a throwaway feature branch; nothing into DEV until the Spike team explicitly graduates the work

Why · rationale for each rule

  • DEV always green: enables continuous delivery and catches regressions immediately. Otherwise DEV silently rots and PROD pushes become risky events.
  • ≤100 LOC PRs: keeps review depth high. Big PRs get rubber-stamped; small PRs get real review, from both agents and humans.
  • ≤5 features per sprint per team: prevents PR queue blowup, forces focused work, keeps DEV's blast radius bounded.
  • Story branches: clean history, atomic units of work, easy to revert, easy to review.
  • Architect approval for exceptions: rules without enforcement become suggestions. The Architect is the enforcement layer.
  • Red team can tag PRs: keeps the discipline honest. If something feels off, anyone can pull it into deeper review.

PR author · checklist

  • ☐ PR is ≤100 LOC
  • ☐ Single concern, not bundled
  • ☐ Tests included
  • ☐ Link to L1 in PR description (so reviewer can check against intent)
  • ☐ Observability included where relevant (no obs plan means no L1 approval, so this should already be covered)
  • ☐ Self-review done (read your own diff before requesting review)
  • ☐ If agent-generated: you understand it. "The agent wrote it" is not a defense.

PR reviewer · checklist

  • ☐ Code matches L1 intent (not just internally consistent)
  • ☐ Edge cases considered
  • ☐ No copy-paste from other parts of codebase (cross-team drift check)
  • ☐ Approve or specific feedback within 4 working hours
  • ☐ If concerned: tag for red team rather than block indefinitely

DEV stays green · the discipline

  • Broken DEV stops the line for all teams using that DEV branch
  • Whoever broke it owns the fix, top priority
  • Other PRs blocked until DEV is green
  • Pattern of repeated breaks → red team review of the underlying area
